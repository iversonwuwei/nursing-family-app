# 家属 APP + AI 工程设计

## 范围

本工程聚焦养老护理场景中的家属端能力，围绕“信任、透明、沟通”三条主线建设。

当前交付范围：

- 老人状态首页
- 健康趋势与解释
- 护理记录与家属反馈
- 探视中心与远程视频
- 消息中心与账单中心
- AI 今日摘要与 AI 探视助手

## 文档来源

设计依据来自 nursing-documents 工程中的以下平台文档：

- 产品设计：定义家属端页面列表、UI 色板与交互流程
- 模块映射：定义家属端应围绕老人状态、护理、探视、账单和 AI 助手构建
- AI Agent 架构：约束 AI 只提供解释、建议、协同，不直接替代业务决策
- 平台架构：约束家属 APP 只访问家属侧可见摘要对象，不污染员工或 Admin 权限模型
- 实施蓝图：建议家属 APP 独立建仓，先做展示闭环，再接真实服务

## 用户影响

- 家属可直观看到老人今日状态和护理执行情况
- 家属可用更少专业术语理解健康趋势
- 家属可直接进入探视、视频沟通、账单与反馈流程
- 家属可通过 AI 摘要和 AI 探视助手降低沟通成本

## 技术设计

### 状态管理

- 使用 GetX 作为路由、依赖注入和页面状态管理方案
- 所有页面统一使用 GetView 创建
- 服务层先落 MockFamilyService，便于后续替换真实 API

### 页面结构

- RootView 提供五个主 Tab：首页、健康、护理、探视、我的
- 次级页面通过 GetPage 管理：消息、账单、反馈、视频、报警历史、AI 页面
- 页面内部通过小组件拆分区块：卡片、趋势、状态角标、快捷入口、时间线

### 设计系统

- 主色采用温暖绿、米色、浅蓝，延续 documents 中的家属端配色原则
- 页面背景保持低刺激米白色，卡片使用大圆角和柔和阴影
- 正文默认 16px 及以上，保持家属友好阅读体验

## UI 状态

- 加载态：当前工程以 Mock 数据为主，后续接入真实接口时统一补骨架屏
- 空态：通过空说明文案和兜底摘要保持页面完整
- 错误态：当前通过次级区块降级实现，不阻断主流程
- 移动端：基于 Flutter 手机端优先布局构建
